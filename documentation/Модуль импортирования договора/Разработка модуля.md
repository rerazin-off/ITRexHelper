# Модуль импортирования договора

## 1. Общее описание модуля

Модуль импортирования договора отвечает за динамическое формирование и скачивание PDF-документов в системе ITRexHelper. Модуль не хранит готовые файлы в базе данных: при каждом запросе сервер собирает актуальные данные из существующих моделей, заполняет DOCX-шаблон и возвращает пользователю PDF-файл.

Модуль реализует два независимых, но архитектурно схожих сценария:

- **Скачивание договора** — формирование договора оказания услуг от лица компании-исполнителя (ООО «Айти Партнер») по завершённой заявке.
- **Скачивание отчёта по аналитике** — формирование PDF-отчёта со статистикой заявок для сотрудников поддержки и администраторов.

Модуль интегрирован с модулем заявок, модулем пользователей и страницей аналитики. Доступ к документам разграничен по ролям пользователей.

---

## 2.6.1. Основные функции модуля

### 2.6.1.1. Основные функции модуля

Модуль предоставляет следующий функционал:

- Формирование PDF-договора по закрытой заявке от лица компании-исполнителя
- Скачивание договора клиентом по своей завершённой заявке
- Скачивание договора сотрудником поддержки и администратором по любой завершённой заявке
- Формирование и скачивание PDF-отчёта по аналитике заявок (сотрудник / администратор)
- Заполнение DOCX-шаблонов с сохранением форматирования документа
- Конвертация DOCX в PDF с резервным механизмом генерации через ReportLab
- Проверка прав доступа и статуса заявки перед выдачей документа
- Информативная обработка ошибок с понятными сообщениями пользователю

### 2.6.1.2. Архитектурное решение

При разработке модуля было принято решение **не сохранять PDF-файлы в базе данных**, а генерировать их по запросу. Такой подход обеспечивает:

- отсутствие необходимости в миграциях и дополнительных полях модели `Ticket`;
- актуальность данных в документе на момент скачивания;
- хранение оформления документа в файловых шаблонах (`document_templates/`), что соответствует диаграмме развёртывания проекта (блок «Шаблоны»).

Архитектура модуля разделена на слои:

| Слой | Файлы | Назначение |
|------|-------|------------|
| Представления | `tickets/document_views.py` | HTTP-запросы, проверка прав, ответ с PDF |
| Сервисы | `tickets/documents/services.py` | Сбор данных из БД (только чтение) |
| Генератор | `tickets/documents/generator.py` | Заполнение шаблона и конвертация в PDF |
| Резервный генератор | `tickets/documents/pdf_fallback.py` | PDF через ReportLab при недоступности конвертера |
| Шаблоны | `document_templates/*.docx` | Исходное форматирование документов |

Для заполнения шаблонов используется библиотека **docxtpl** (Jinja2-подстановки внутри Word-документа). Для конвертации применяется **docx2pdf** (Windows + Microsoft Word) или **LibreOffice** в headless-режиме; при их недоступности — **ReportLab**.

## 2.6.2. Модель данных (Models)

### 2.6.2.1. Принцип работы с данными

Модуль **не вводит собственных моделей Django** и не изменяет существующую схему базы данных. Все данные для документов берутся из уже существующих сущностей:

**Модель заявки (`Ticket`):**

- идентификатор, описание, статус, дата создания;
- контактная информация;
- связь с автором (клиентом) и исполнителем (сотрудником).

**Модель пользователя (`CustomUser`):**

- фамилия, имя, отчество;
- название компании, email, телефон;
- роль пользователя (CLIENT / STAFF / ADMIN).

**Модель истории статусов (`TicketStatusHistory`):**

- используется при формировании отчёта по аналитике (журнал аудита).

### 2.6.2.2. Контекст договора

Функция `collect_contract_context(ticket)` формирует словарь подстановок для шаблона:

| Поле | Источник | Пример |
|------|----------|--------|
| `contract_number` | ID заявки | `ITR-00042` |
| `contract_date` | Текущая дата | `23.06.2026` |
| `contractor_name` | Константа модуля | `ООО «Айти Партнер»` |
| `contractor_details` | Константа модуля | ИНН, адрес |
| `client_company` | `ticket.author.company_name` | `ООО Тест` |
| `client_name` | ФИО автора заявки | `Иванов Иван Иванович` |
| `client_contact` | `contact_info` / телефон / email | `+79991234567` |
| `ticket_id`, `ticket_date`, `ticket_description`, `ticket_status` | Модель `Ticket` | — |
| `executor_name` | `ticket.executor` | `Петров Пётр` |

### 2.6.2.3. Контекст отчёта по аналитике

Функция `collect_analytics_context()` повторяет логику страницы `/tickets/analytics/`:

- общее количество заявок (без служебных тикетов «Чат поддержки»);
- заявки в работе, просроченные, за текущий месяц;
- динамика за 6 месяцев;
- последние 20 записей журнала аудита.

### 2.6.2.4. Связи с другими моделями

Модуль связан с другими сущностями системы:

- **Заявки** — источник данных для договора;
- **Пользователи** — данные клиента и проверка роли;
- **История статусов** — данные для PDF-отчёта;
- **Файловые шаблоны** — хранение оформления документов вне БД.

## 2.6.3. Шаблоны документов (Templates)

### 2.6.3.1. Шаблон договора (`contract.docx`)

DOCX-шаблон содержит структурированный текст договора оказания услуг:

- заголовок с номером договора;
- дата и место заключения;
- стороны: Исполнитель (компания ITREX / Айти Партнер) и Заказчик (клиент);
- предмет договора — описание услуг по заявке;
- реквизиты сторон;
- блок подписей.

Placeholders заполняются через `docxtpl` без нарушения исходного форматирования Word-документа.

### 2.6.3.2. Шаблон отчёта (`analytics_report.docx`)

Содержит:

- заголовок «IT REX HELPER — Отчёт по аналитике заявок»;
- дату формирования;
- таблицу сводных показателей;
- блок динамики за 6 месяцев;
- журнал аудита изменений статусов.

### 2.6.3.3. Пересоздание шаблонов

```bash
cd site/ITRexHelper
python manage.py shell -c "from tickets.documents.build_templates import create_contract_template, create_analytics_template; create_contract_template(); create_analytics_template()"
```

## 2.6.4. Представления и бизнес-логика (Views)

### 2.6.4.1. Скачивание договора (`download_ticket_contract`)

**URL:** `/tickets/<ticket_id>/contract/download/`  
**Файл:** `tickets/document_views.py`

Алгоритм работы:

**Этап 1.** Пользователь инициирует скачивание (кнопка «Скачать договор»).

**Этап 2.** Сервер загружает заявку по ID.

**Этап 3.** Проверка прав доступа:

| Роль | Условие |
|------|---------|
| CLIENT | Только своя заявка |
| STAFF / ADMIN | Любая заявка |
| Неавторизованный | Редирект на вход |

**Этап 4.** Проверка статуса: договор доступен **только для заявок со статусом CLOSED (Закрыта)**. Для всех ролей, включая администратора.

**Этап 5.** Вызов `collect_contract_context(ticket)` — сбор данных.

**Этап 6.** Заполнение `contract.docx` и конвертация в PDF.

**Этап 7.** Ответ `Content-Disposition: attachment; filename="contract_ITR-XXXXX.pdf"`.

**Точки входа в интерфейсе:**

- Клиент: модальное окно «Подробнее» на странице «Мои заявки» (только закрытые заявки).
- Сотрудник / администратор: модальное окно заявки на странице «Все заявки».
- Сотрудник / администратор: страница детального просмотра заявки (`/tickets/<id>/`).

### 2.6.4.2. Скачивание отчёта по аналитике (`download_analytics_report`)

**URL:** `/tickets/analytics/download/`  
**Файл:** `tickets/document_views.py`

Алгоритм работы:

**Этап 1.** Сотрудник или администратор нажимает «Скачать отчет» на странице аналитики.

**Этап 2.** Проверка роли: доступ только для STAFF и ADMIN.

**Этап 3.** Вызов `collect_analytics_context()`. Если заявок нет — ошибка «Нет данных о заявках для формирования отчёта».

**Этап 4.** Заполнение `analytics_report.docx` и конвертация в PDF.

**Этап 5.** Скачивание файла `analytics_report.pdf`.

### 2.6.4.3. Генератор PDF (`generate_pdf_from_template`)

**Файл:** `tickets/documents/generator.py`

1. Загрузка DOCX-шаблона из `document_templates/`.
2. Рендеринг через `docxtpl.DocxTemplate.render(context)`.
3. Конвертация DOCX → PDF (`docx2pdf` / LibreOffice).
4. При ошибке конвертации — вызов `pdf_fallback.render_pdf_fallback()`.

## 2.6.5. Клиентская часть (JavaScript)

### 2.6.5.1. Скачивание договора клиентом

**Файл:** `static/js/app.js`

При открытии модального окна заявки (`data-ticket-open`):

- если статус заявки `CLOSED`, в footer модального окна динамически добавляется ссылка «Скачать договор»;
- ссылка ведёт на `/tickets/<id>/contract/download/`;
- для незакрытых заявок кнопка скрыта.

### 2.6.5.2. Скачивание договора администратором и сотрудником

**Файл:** `static/js/app.js`

При открытии модального окна заявки в админ-панели (`data-admin-ticket-open`):

- передаётся `data-ticket-id` и `data-status-value`;
- для статуса `CLOSED` в footer модального окна отображается «Скачать договор»;
- на странице детального просмотра заявки кнопка отображается через шаблон `ticket_detail.html`.

Используются существующие CSS-классы (`btn`, `btn-outline`, `btn-sm`) без изменения общего дизайна системы.

## 2.6.6. Безопасность

**Аутентификация.** Все эндпоинты защищены декоратором `@login_required`. Неавторизованный пользователь перенаправляется на страницу входа.

**Авторизация по ролям.**

- Клиент скачивает только свой договор.
- Сотрудник и администратор скачивают договор любой заявки, но только в статусе «Закрыта».
- Отчёт по аналитике недоступен клиентам.

**Проверка статуса заявки.** Договор не формируется для заявок в статусах NEW, IN_PROGRESS, WAITING, REJECTED, CANCELED.

**CSRF-защита.** Прямые GET-запросы на скачивание не требуют CSRF-токена (идempotent read-only операция с проверкой прав). Формы редактирования заявок по-прежнему защищены CSRF.

**Отсутствие хранения файлов на сервере.** PDF не сохраняется на диск и не попадает в медиа-хранилище, что снижает риск несанкционированного доступа к чужим документам через прямую ссылку.

## 2.6.9. Заключение

Модуль импортирования договора обеспечивает:

- формирование юридически оформленных PDF-договоров от лица компании-исполнителя по завершённым заявкам;
- возможность скачивания договора клиентом, сотрудником поддержки и **администратором** с разграничением прав;
- формирование PDF-отчётов по аналитике для административного персонала;
- сохранение форматирования документов через DOCX-шаблоны;
- работу без изменений схемы базы данных;
- полное покрытие ключевых сценариев unit-тестами.

Модуль соответствует диаграммам деятельности, последовательности, коммуникации и состояний из папки `documentation/Модуль импортирования договора/`.
